# Требования к модулю `forza_core` (Версия 1.0)

## 1. Функциональные требования (Что модуль должен делать)

### Сетевой ввод-вывод (Receiving)
Модуль должен уметь запускать прослушивание заданного UDP-порта для получения датаграмм от игры Forza.

> **Заметка Архитектора - Безопасность:**  
> Порт должен жестко биндиться на `127.0.0.1` (localhost), если игра и приложение на одном ПК. Это закроет вектор атак типа UDP Spoofing из внешней сети.

> **Заметка Архитектора - Актуальность:**  
> Реализовывать сетевой слой рекомендуется через `asyncio` (например, `asyncio.DatagramProtocol`), а не через блокирующие сокеты.

### Базовая фильтрация пакетов (Filtering)
Модуль должен отбрасывать невалидные сетевые пакеты на самом раннем этапе.  
Критерии: пустой пакет, пакет неверного размера (не соответствующий структуре телеметрии Форзы).

> **Заметка Архитектора - Безопасность:**  
> Размер пакета (`length`) должен проверяться строго до передачи в парсер/десериализатор во избежание бинарных уязвимостей и ошибок переполнения.

### Парсинг / Десериализация (Parsing)
Модуль должен конвертировать валидный массив байт (`payload`) в удобную структуру данных (Data Transfer Object — DTO), содержащую понятные типы данных (`float`, `int`) и названия полей (скорость, обороты, номер передачи и т.д.).

> **Заметка Архитектора - SOLID:**  
> Для поддержки разных версий игры (Forza Horizon 4, 5, Motorsport) используй паттерн "Стратегия" или "Фабрика". Это позволит добавлять новые парсеры, не меняя существующий код (Open/Closed Principle).

> **Заметка Архитектора - Профессиональность:**  
> Имена полей в DTO должны явно содержать единицы измерения: `speed_kmh` (или `mps`), `engine_rpm`, `gear_number`.

### Санитарная валидация данных (Sanity Check)
Проверка распакованных значений на "здравый смысл" перед отдачей.  
Примеры: скорость не может быть меньше 0, номер передачи не может быть -5 и т.д.

> **Заметка Архитектора - Безопасность:**  
> Помимо минусовых значений, обязательно проверяй `float` на `NaN` (Not a Number) и `Infinity`. Битые данные с `NaN` могут сломать UI или сериализацию в JSON на стороне клиента.

> **Заметка Архитектора - Актуальность:**  
> Используй библиотеку `Pydantic` для DTO. Она позволит описать валидацию декларативно (например, `speed_kmh: float = Field(ge=0.0)`).

### Экспорт данных (Delivery)
Модуль должен отдавать проверенный DTO-объект приложению-потребителю (хосту/UI).

> **Заметка Архитектора - SOLID:**  
> Модуль не должен вызывать хост напрямую (Dependency Inversion). Используй паттерн Observer (события/подписки) или потокобезопасные очереди (например, `asyncio.Queue`), чтобы хост сам забирал данные.

### Управление состоянием (Lifecycle)
Модуль должен предоставлять интерфейс управления: возможность запустить процесс сбора (`start`) и безопасно его остановить (`stop`), освободив сетевой порт.

> **Заметка Архитектора - Профессиональность:**  
> Методы `start()` и `stop()` должны быть идемпотентными. Повторный вызов `start()` не должен крашить приложение ("Port in use"), а повторный `stop()` не должен вызывать ошибок.

> **Заметка Архитектора - SOLID:**  
> Раздели интерфейсы (Interface Segregation). Сделай отдельный `ILifecycle` (start/stop) и отдельный `IDataProvider` (получение данных/подписка).

## 2. Нефункциональные требования (Как модуль должен работать)

### Изолированность потока выполнения
Модуль не должен блокировать основной поток вызывающего приложения.  
Цикл прослушивания UDP-порта должен работать асинхронно или в отдельном потоке.

> **Заметка Архитектора - Профессиональность:**  
> Обязательно обеспечь потокобезопасность. Передача DTO из фонового слушателя в основной поток через обычные переменные вызовет состояние гонки (Race Condition). Используй очереди (`queue.Queue` / `asyncio.Queue`).

### Отказоустойчивость (Fault Tolerance)
Модуль не должен завершаться с критической ошибкой (crash), если пришёл битый или непредвиденный пакет.  
Ошибка должна глушиться (или логироваться), а модуль должен продолжить ждать следующий пакет.

> **Заметка Архитектора - Безопасность:**  
> Внедрить Rate Limiting для логов. При шквале битых пакетов логгер может забить жесткий диск или упереться в I/O узкое горлышко (например, писать максимум 1 ошибку парсинга в секунду).

### Единая ответственность (SRP - Single Responsibility)
Модуль является исключительно "поставщиком" данных (Provider). Он не знает о базах данных, HTTP-запросах на бэкенд и пользовательском интерфейсе.

> **Заметка Архитектора - SOLID:**  
> Этот принцип нужно спустить и на уровень самого модуля. Внутри `forza_core` логика должна быть разбита на независимые классы:  
> - `UdpListener` (сеть)  
> - `PacketParser` (байты → DTO)  
> - `DataValidator` (проверка логики)
