## Архитектурный паттерн: MVVM (Model-View-ViewModel)

Весь UI строится на строгом разделении ответственности в соответствии с паттерном MVVM:

*   **View (Представление):** 
    *   Отвечает **только** за отображение данных, реакцию на визуальные события и сбор ввода пользователя.
    *   Сам UI рисуется в Qt Designer (`.ui` файлы), которые затем компилируются в Python-код (`ui_*.py`).
    * **Никогда не редактируем** сгенерированные файлы вручную.
    *   Логика View ограничена подпиской на события ViewModel и передачей действий пользователя (например, нажатие кнопок) во ViewModel.
    *   Связывание Qt-виджетов с доменными моделями вынесено в мапперы (например, `TuningMapper`), которые используют строгий декларативный подход `WidgetBinding` для синхронизации.
*   **ViewModel (Модель Представления):**
    *   Агрегирует данные из доменного слоя и подготавливает их для отображения во View.
    *   Не зависит от конкретных Qt-виджетов или сигналов Qt (поверхностно), что облегчает тестирование. Оперирует абстрактными структурами данных.
    *   Обрабатывает бизнес-логику UI (вызовы сохранения, статусы, ошибки валидации, подготовку словарей) и оповещает View через систему сигналов (например, `config_saved`, `validation_failed`).
*   **Model (Модель):**
    *   Слой доменной логики полностью изолирован от представления. Интерфейс взаимодействует с `ConfigStateManager` и другими сервисами сугубо через ViewModel. **Важно:** ViewModel получает сервисы через инъекцию зависимостей (Dependency Injection), а не создает их сама, что гарантирует соблюдение принципа DIP (Dependency Inversion Principle) на уровне UI.

## Общие правила работы с Qt и UI

1.  **Запрет ручного редактирования кода View:** Мы **никогда не редактируем** сгенерированные Python-файлы (`ui_*.py`) вручную. Любые изменения в верстке вносятся исключительно через Qt Designer в исходных `.ui` файлах.
2.  **Декларативный маппинг вместо ручного:** Ручное связывание каждого виджета (сигналы/слоты/геттеры) запрещено для форм с данными. Используются централизованные мапперы на основе `WidgetBinding`. Конфигурация включает пути к полям Pydantic-модели и функции-трансформеры (`Transformers` из `transformers.py`).
3.  **Неявное закрытие диалогов:** Стандартные сигналы кнопок `QDialogButtonBox` (вроде `accepted()` или `rejected()`) отключаются. View не имеет права проактивно закрыть окно по нажатию "Сохранить", пока ViewModel не подтвердит успешную валидацию и сохранение.
4.  **Обработка ошибок:** В случае ошибки валидации от доменного слоя (например, `Pydantic ValidationError`), окно не зарывается. View получает список ошибок и самостоятельно применяет визуальную индикацию (например, метод `highlight_errors()` добавляет красную рамку к неверным виджетам).
5.  **Разделение на `.ui` и `.py`:** Источником истины для визуального расположения во всем проекте всегда выступает XML-файл в папке `src/desktop_client/presentation/assets/`.

## Карта окон (Текущие и Планируемые)

Текущая экосистема окон приложения делится на несколько основных сущностей:

### 1. MainWindow (Главное окно)
*   **Статус:** В разработке / Задел на будущее.
*   **Роль:** Основная панель управления.
*   **Ожидаемый функционал:**
    *   Старт/Стоп сессии сбора телеметрии.
    *   Живые графики, дэшборды или индикаторы статуса (состояние синхронизации, счетчик буфера).
    *   Окно входа (логин) или навигация по остальным разделам приложения.

### 2. ConfigDialog (Окно конфигурации)
*   **Статус:** Активно, полностью реализует новую архитектуру MVVM.
*   **Роль:** Ввод параметров автомобиля (давление шин, аэродинамика, подвеска) перед заездом.
*   **Особенности:** Декларативные биндинги (`tuning_binder.py`), строгая валидация Pydantic-модели `TuningSetup`, отсутствие прямого доступа к файловой системе из UI (сохранение идет только через `ConfigStateManager`).

### 3. SettingsDialog (Настройки приложения / Мини-окно)
*   **Статус:** Заглушка, архитектура проектируется (см. `04_Domain_Logic/02_settings_window.md`).
*   **Роль:** Изменение системных параметров: аутентификация, директории, URL бэкенда, параметры локального буфера базы.
*   **Особенности:** Ожидается, что мини-окно будет позволять применять "быстрые настройки" на лету без сложной перезагрузки, взаимодействуя со своим собственным стейтом.

### 4. SessionsLibrary (Библиотека заездов)
*   **Статус:** Задел на будущее.
*   **Роль:** Просмотр истории заездов.
*   **Ожидаемый функционал:** Простой список сессий с базовыми фильтрами и возможностью выбора конкретного заезда для детального ознакомления.

### 5. AnalyzerWindow (Анализатор)
*   **Статус:** Задел на будущее.
*   **Роль:** Глубокий анализ данных.
*   **Ожидаемый функционал:** Визуализация телеметрии заезда, графики и инструменты для сравнения показателей производительности.
