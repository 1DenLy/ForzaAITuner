# Руководство разработчика (UI Development Guidelines)

Это руководство содержит инструкции по работе с пользовательским интерфейсом (UI) проекта ForzaAITuner. Все разработчики должны следовать этим правилам для обеспечения консистентности кода и предотвращения поломок.

---

## 🛠 Первичная настройка

Перед началом работы с UI (один раз после клонирования репозитория) выполните команду:

```powershell
make setup-hooks
```

Это установит `pre-commit` и зарегистрирует git-хук. После этого компиляция `.ui → .py` будет запускаться **автоматически** при каждом `git commit`, если в закоммиченных файлах есть изменения в `.ui`.

---

## 🚫 Запреты и ограничения

### ❌ Не редактируйте `.ui`-файлы вручную
`.ui`-файлы — это XML-скрипты, которые редактируются **только в Qt Designer**. Логика Qt Designer сложна, и ручное редактирование XML может привести к тому, что файл перестанет открываться или будет генерировать некорректный Python-код.

**ЗАПРЕЩЕНО:** Открывать `.ui` файлы в папке `src/desktop_client/presentation/assets/` как текстовые файлы и изменять их вручную. Источником истины для визуального расположения во всем проекте всегда выступает XML-содержимое этих файлов, редактируемое через Qt Designer.
**ПРАВИЛЬНО:** Открывать файлы только через Qt Designer.

### ❌ Не редактируйте сгенерированные Python-файлы
Файлы в директории `ui_gen` (например, `ui_config_dialog.py`) генерируются автоматически командой `pyside6-uic`. Любые ручные правки в этих файлах будут **безвозвратно удалены** при следующей регенерации.

**ЗАПРЕЩЕНО:** Редактировать файлы вручную в `src/desktop_client/presentation/ui_gen/`.
**ПРАВИЛЬНО:** Использовать специальный скрипт для генерации (или довериться pre-commit хуку):
```powershell
python scripts/compile_ui.py
```

### ❌ Не используйте "магические строки" (Hardcode)
Все строки, которые видит пользователь (заголовки окон, сообщения об ошибках, тексты уведомлений), должны быть вынесены в централизованное хранилище ресурсов. Это необходимо для обеспечения консистентности UI и будущей локализации приложения.

**ЗАПРЕЩЕНО:** Писать текст напрямую в вызовах `QMessageBox`, `QFileDialog` или `setText()`.
**ПРАВИЛЬНО:** Использовать константы из `desktop_client.presentation.resources.strings.UIStrings`.

## 📋 Сценарии изменений (How-to)

### Сценарий 1: Добавить новый слайдер (×10 scale, float)
*Применяется для:* давление шин, развал, схождение, кастор.

1. Добавьте `horizontalSlider_<name>` и `label_<name>_value` в `.ui` через Qt Designer.
2. Перегенерируйте Python-код.
3. Добавьте поле в модель домена (напр. `tuning.py`) с валидацией `Field(ge=..., le=...)`.
4. Добавьте `WidgetBinding` в соответствующий маппер (напр. `TuningMapper._init_bindings()`):
   ```python
   WidgetBinding("horizontalSlider_<name>", ("<section>", "<field>"),
       ui_to_model=Transformers.slider_x10_to_model,
       model_to_ui=Transformers.slider_x10_to_ui,
       label_name="label_<name>_value",
       label_format="float"),
   ```
5. Добавьте значение по умолчанию в `TuningDefaults.create()`.

### Сценарий 2: Добавить новый QDoubleSpinBox
*Применяется для:* подвеска, демпфирование и любые float-поля.

1. Добавьте `doubleSpinBox_<name>` в `.ui`.
2. Перегенерируйте Python-код.
3. Добавьте поле в модель домена.
4. Добавьте `WidgetBinding` (трансформаторы не нужны, используется identity по умолчанию):
   ```python
   WidgetBinding("doubleSpinBox_<name>", ("<section>", "<field>")),
   ```

### Сценарий 3: Добавить новый QComboBox
1. Добавьте `comboBox_<name>` в `.ui`.
2. Перегенерируйте Python-код.
3. Добавьте варианты значений в `TuningOptions.for_path()` (файл `tuning_options.py`).
4. Добавьте `WidgetBinding`:
   ```python
   # Строковые значения — трансформация не нужна:
   WidgetBinding("comboBox_<name>", ("<section>", "<field>")),
   ```

### Сценарий 4: Переименовать виджет (objectName)
1. Найдите старое имя в файле биндингов (напр. `tuning_binder.py`) — используйте поиск (grep) по `"старое_имя"`.
2. Замените `widget_name=` в соответствующем `WidgetBinding`.
3. Если есть `label_name=` — обновите и его.

## ✅ Чеклист для проверки (перед коммитом)

- `ui_*.py` содержит `objectName`, совпадающий с `widget_name` в `WidgetBinding`.
- Каждый `QSlider` имеет `WidgetBinding` с `ui_to_model`/`model_to_ui` трансформерами.
- Каждый слайдер с live-лейблом имеет `label_name` и `label_format` в своём `WidgetBinding`.
- `model_path` в каждом `WidgetBinding` соответствует реальному полю в модели домена.
- Новое поле домена добавлено в метод создания дефолтных настроек (напр. `TuningDefaults.create()`).
- Сигналы `validation_failed`, `config_saved`, `global_error_occurred` и `preset_loaded` подключены во View.

---

## 🔄 Жизненный цикл сохранения (Save Flow)

1. **Клик Save:** Срабатывает слот `_on_save_clicked()` во View.
2. **Сбор данных:** View вызывает `self.mapper.update_from_ui()` для получения сырых данных из виджетов.
3. **Валидация:** ViewModel передает данные в `ConfigValidatorService` для проверки через Pydantic.
4. **Результат:**
   - **УСПЕХ:** Модель сохраняется, испускается сигнал `config_saved`, окно закрывается.
   - **ОШИБКА:** Испускается `validation_failed(errors)`. View вызывает `self.mapper.highlight_errors()` для подсветки проблемных полей красным. Окно остается открытым.

---

## 🔄 Жизненный цикл загрузки пресетов (Load Preset Flow)

1. **Клик Open:** Пользователь нажимает кнопку открытия файла, срабатывает слот `_on_open_preset_clicked()` во View (`ConfigDialog`), который открывает диалог выбора файла.
2. **Передача пути:** Выбранный путь передается во ViewModel через абстракцию `IConfigViewModel.load_config_from_file(filepath)`.
3. **Чтение I/O:** ViewModel делегирует чтение абстракции `IPresetRepository.load_preset(filepath)` (инкапсулирующей проверки пути и размера).
4. **Валидация Pydantic:** ViewModel парсит JSON-текст через `TuningSetup.model_validate_json(text)`.
5. **Результат:**
   - **УСПЕХ:** Модель распарсена, испускается сигнал `preset_loaded(dict)`. View обновляет виджеты через маппер и показывает уведомление об успехе.
   - **ОШИБКА:** Испускается `global_error_occurred(msg)`. View показывает окно с критической ошибкой.
