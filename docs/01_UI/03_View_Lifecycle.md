# 03. Жизненный цикл представлений (View Lifecycle)

Этот документ описывает стандартный жизненный цикл любого окна (View) в проекте, от его создания до закрытия. Строгое следование этому циклу гарантирует предсказуемое поведение UI, правильный биндинг данных и корректную обработку состояний.

## 1. Универсальный порядок инициализации

При создании нового окна или диалога (например, `MainWindow`, `ConfigDialog`, `SettingsDialog`), инициализация должна проходить в строго определенном порядке. Это предотвращает ситуации, когда сигналы срабатывают до того, как загружены данные или настроены мапперы (что может привести к нежелательным side-эффектам).

Порядок инициализации (обычно реализуется в `__init__` или разделяется на вспомогательные методы `_setup_*`):

1. **Setup UI (`setupUi`)**: 
   - Вызов автосгенерированного кода из `.ui` файла (например, `self.ui.setupUi(self)`).
   - Базовая настройка виджетов, которая не может быть выполнена в Qt Designer.

2. **Инициализация мапперов (Mappers / Binders)**:
   - Настройка `WidgetBinding`, `TuningMapper`, трансформеров и других связующих компонентов.
   - Определение того, как конкретные виджеты будут отображать и собирать данные.
   - *Важно: на этом этапе сами данные еще не загружаются в виджеты.*

3. **Подключение сигналов (Connect Signals)**:
   - Привязка обработчиков к кнопкам (Save, Cancel, Apply).
   - Подключение событий изменения виджетов (например, `textChanged`, `valueChanged`) к внутренней логике.
   - Подписка на события из глобального состояния (события от слоя ViewModel, контроллеров или SignalBus).

4. **Загрузка данных (Load Data / Initial Update)**:
   - Получение начального состояния из ViewModel, дефолтных значений (`TuningDefaults`) или текущей конфигурации.
   - Первичное заполнение виджетов реальными данными.
   - Вызов первоначального обновления UI (например, `self.mapper.update_ui(initial_data)`).

### Пример стандартного шаблона `__init__`

```python
class MyCustomDialog(QDialog):
    def __init__(self, view_model, parent=None):
        super().__init__(parent)
        self.view_model = view_model
        
        # 1. Setup UI
        self.ui = Ui_MyCustomDialog()
        self.ui.setupUi(self)
        
        # 2. Мапперы
        self._setup_bindings()
        
        # 3. Коннект сигналов
        self._connect_signals()
        
        # 4. Загрузка данных
        self._load_initial_data()
```

## 2. Правила обработки закрытия окон

Важным аспектом жизненного цикла является правильная обработка закрытия диалоговых окон. Стандартное поведение Qt (где кнопки в `QDialogButtonBox` автоматически вызывают `accept()` или `reject()`) часто не подходит для нашего проекта, так как перед закрытием требуется выполнить валидацию данных или асинхронное сохранение.

### Отмена стандартного accept/reject

Для диалогов, требующих валидации перед закрытием (например, редактирование настроек или конфигурации):

1. **Не используйте автоматические сигналы `accepted` / `rejected`** (чтобы окно не закрывалось сразу после клика на кнопку "OK").
2. Вместо этого, перехватывайте нажатие кнопок самостоятельно:
   - Подключите кнопку "Save" к собственному обработчику (например, `on_save_clicked`).
   - При использовании `QDialogButtonBox` переопределите поведение кнопок, не связывая их напрямую с `accept()` в Qt Designer.

#### Привязки кнопок управления (QDialogButtonBox)

В `ConfigDialog` нижняя панель кнопок (`QDialogButtonBox`) настроена вручную для четкого следования принципам SOLID. Стандартные сигналы `accepted()` и `rejected()` окна отключены от кнопок по умолчанию, чтобы избежать автоматического закрытия без валидации.

| Кнопка | Тип | Сигнал UI | Метод View (маршрутизатор) | Вызываемый метод / сигнал ViewModel |
| :--- | :--- | :--- | :--- | :--- |
| **Save** | `QDialogButtonBox.Save` | `clicked` | `_on_save_clicked()` | `vm.apply_config(raw_data)` |
| **Reset** | `QDialogButtonBox.Reset` | `clicked` | `_load_last_valid_config()` | `vm.get_last_valid_config()` |
| **Close** | `QDialogButtonBox.Close` | `clicked` | `self.reject()` | — (закрытие без сохранения) |
| **Open** | `QDialogButtonBox.Open` | `clicked` | `_on_open_preset_clicked()` | — (заглушка, в разработке) |

> **Важно:** Закрытие окна по кнопке **Save** происходит *только* при получении сигнала `config_saved` от ViewModel. Окно само себя закрывает через вызов `self.accept()`.

### Процесс валидации и успешного закрытия (Save Flow)

1. **Сбор данных**: При нажатии на кнопку сохранения, данные собираются из формы (часто через мапперы, например `data = self.mapper.get_data()`).
2. **Валидация**: Собранные данные передаются в валидатор или ViewModel (проверка Pydantic моделями, валидация бизнес-логики).
3. **Обработка результата**:
   - **Успех**: Если данные валидны и сохранены, диалог закрывается программно вызовом метода `self.accept()`. Главное окно получает код `QDialog.Accepted` и может реагировать соответствующим образом.
   - **Ошибка**: Диалог **не закрывается**. Во View отображается сообщение об ошибке (выделение красным цветом ошибочных полей, показ `QMessageBox`, текст ошибки в статусе), позволяя пользователю исправить ввод.

### Обработка отмены (Cancel Flow)

1. При нажатии кнопки "Cancel" или закрытии окна (через крестик или событие `closeEvent`), все внутренние несохраненные изменения отклоняются.
2. Для явного закрытия с отменой вызывается `self.reject()`.
3. В некоторых случаях (если форма была изменена), перед вызовом `reject()` может быть полезно запросить подтверждение пользователя ("У вас есть несохраненные изменения. Вы уверены, что хотите выйти?").
